# Monica Unified Module Architecture

## Status

- Draft v2
- Scope: `Monica.*` infrastructure modules, `Monica.*.UI` UI modules, Mixed modules
- This document defines the standard directory structure, dependency rules, and responsibility boundaries for all Monica modules
- When legacy conventions conflict with this document, this document takes precedence

## Goals

This specification addresses the following long-term problems:

- Prevent bloated god-services like `RAGService` that accumulate all business logic, indexing, search, provider coordination, and state recovery in a single file
- Establish clear boundaries between public API surface (Facades, Abstractions, Models) and internal implementation (Services, Providers)
- Unify the structure of infrastructure modules, UI modules, and mixed modules under one consistent pattern
- Ensure new modules can be scaffolded with a stable, predictable directory layout

## Core Principles

### 1. Feature-First, Layers Inside Features

Monica does not use a flat project-level `Services/Models/Abstractions` layout.

The correct approach:

- Split by feature at the project level first
- Use standard layer folders (`Abstractions/`, `Models/`, `Services/`, `Providers/`, `Facades/`) inside each feature

This is **Feature-First + Layer-Inside-Feature**. It keeps things organized as modules grow, and prevents any single folder from becoming a dumping ground.

### 2. Facades Are the Only Public Entry Point for API and UI

Every module exposes its capabilities through `Facades/` вЂ” thin orchestration services that:

- Return `Res` / `Res<T>` (the unified response model)
- Are consumed by Minimal API endpoints and UI components (directly or via UI modules)
- Delegate to internal `Services/` for actual implementation
- Must NOT contain substantial business logic

Facades are defined in the **infrastructure module**, not in the UI module. This means:

- Minimal API and UI share the same Facade
- UI modules inject the infrastructure Facade directly вЂ” no separate UI Service layer needed for data access
- Other infrastructure modules do NOT consume Facades вЂ” they depend on `Abstractions/` interfaces instead

```
Minimal API  в”Ђв”Ђв†’  Facade (Res<T>)  в”Ђв”Ђв†’  Services (internal)
UI Component в”Ђв”Ђв†’  Facade (Res<T>)  в”Ђв”Ђв†’  Services (internal)
Other Module в”Ђв”Ђв†’  Abstractions (interfaces)  в†ђв”Ђв”Ђ Services implement
```

### 3. Public vs Internal Boundary

Every module has an explicit boundary between what is exposed and what is hidden:

**Public surface** (consumed by external modules, API, UI):
- `Abstractions/` вЂ” interfaces for cross-module dependency
- `Models/` вЂ” shared data contracts
- `Facades/` вЂ” `Res<T>` entry points for API/UI

**Internal implementation** (hidden inside the module):
- `Abstractions/Internal/` вЂ” internal-only contracts
- `Models/Internal/` вЂ” internal-only data types
- `Services/` вЂ” all implementation logic
- `Providers/` вЂ” pluggable strategy implementations

The `Internal/` sub-folder convention makes this boundary visible in the directory structure. Types in `Internal/` should never be referenced by external modules.

### 4. Dependency Direction

Allowed:

```
UI Page / Minimal API
    в†’ Facade (Res<T>, public entry point)
    в†’ Services (internal implementation)
    в†’ Providers / Stores / SDK Adapters
```

Cross-module:

```
Other Module в†’ Abstractions (interfaces) + Models (public)
```

Forbidden:

- Service в†’ Facade (internal must not depend on the public entry point)
- Provider в†’ Facade
- Provider в†’ UI
- Model в†’ Service
- Module registration в†’ business implementation details

### 5. Provider Is a Separate Layer

Providers must be explicitly separated from Services. They are not "special services".

Provider responsibilities:
- Encapsulate vendor SDKs, external systems, file I/O, databases, vector stores
- Implement interfaces defined in `Abstractions/`
- Isolate implementation differences behind a stable contract

Providers do NOT:
- Handle UI-friendly error formatting
- Orchestrate business workflows
- Manage page state
- Return `Res<T>`

### 6. Modules/ Is Registration Only

`Modules/Module{Name}.cs` contains only:
- `Module{Name}`, `Module{Name}Option`, `Module{Name}Guide`, `Module{Name}BuilderExtensions`
- Dependency declarations (`ClaimDependencies`)
- DI registrations (`ConfigureServices`)
- Endpoint mapping registration

`Modules/` must NOT contain business algorithms, API handler logic, UI interaction logic, or provider details.

### 7. Utils Is the Unified Utility Folder

When a module needs pure utility functions, extension methods, or helper classes that don't fit into any feature, use `Utils/`.

- `Utils/` replaces all of: `Helpers/`, `Tools/`, `Utilities/`, `Common/`, `Misc/`
- Only one utility folder name is allowed: `Utils/`
- If a utility class can be attributed to a specific feature, put it inside that feature's folder instead

Exception: `Tools/` is reserved exclusively for AI tool providers (e.g., `KnowledgeSearchToolProvider`) in Monica.AI modules, as these are a distinct concept from general utilities.

## Standard Layer Names

These are the only allowed layer folder names in Monica modules:

| Folder | Purpose | Visibility |
|--------|---------|------------|
| `Abstractions/` | Interfaces, abstract base classes | Public (external modules depend on these) |
| `Abstractions/Internal/` | Internal-only contracts | Private (module-internal only) |
| `Models/` | Records, DTOs, enums, value objects | Public (shared data contracts) |
| `Models/Internal/` | Internal-only data types | Private (module-internal only) |
| `Facades/` | Thin orchestration, returns `Res<T>` | Public (API + UI entry point) |
| `Services/` | Implementation logic | Private (internal to module) |
| `Providers/` | Pluggable strategy implementations | Private (internal to module) |
| `Modules/` | Module registration classes | Public (registration API) |
| `Extensions/` | Extension methods | Public or private depending on usage |
| `Events/` | Domain/integration events | Public (consumed by event handlers) |
| `Exceptions/` | Module-specific exception types | Public (thrown to callers) |
| `Utils/` | Pure utility functions | Private (module-internal) |

### Feature Sub-Folders

When a layer has many files that naturally group by concern, create sub-folders within the layer:

```
Services/
в”њв”Ђв”Ђ Indexing/

```

This applies to all layers: `Abstractions/`, `Models/`, `Services/`, `Providers/`, etc.

### Layer Responsibilities

| Folder | What belongs here | What does NOT belong here |
|--------|-------------------|--------------------------|
| `Modules/` | Module, Option, Guide, BuilderExtensions, DI registration | Business logic, provider details |
| `Abstractions/` | Public interfaces, strategy contracts, store contracts | Implementation classes, UI DTOs |
| `Abstractions/Internal/` | Internal contracts not meant for external consumption | Anything external modules need |
| `Models/` | Public business models, state models, result models, config models | Page ViewModels, browser state |
| `Models/Internal/` | Internal data types (vector records, queue items, etc.) | Anything external modules need |
| `Facades/` | Thin orchestration returning `Res<T>`, parameter normalization | Substantial business logic, SDK calls |
| `Services/` | Single-responsibility business services | `Res<T>` returns, UI adaptation |
| `Services/Support/` | Registry, Resolver, Coordinator, Policy, Normalizer, Factory | Public-facing facade logic |
| `Providers/` | File/DB/vector/SDK/vendor implementations | Business orchestration, UI logic |

### Facade Rules

Facades are the public entry point for API and UI consumers.

Rules:
- Return `Res` / `Res<T>` exclusively
- Stay lightweight вЂ” delegate to `Services/` for implementation
- One facade method typically calls 1-3 internal services
- Handle parameter normalization, use-case orchestration, and error-to-Res conversion
- A single facade file should not exceed ~200 lines вЂ” split into multiple facades by feature if needed

Naming: `{Feature}Facade.cs` (e.g., `RAGFacade.cs`, `AIChatFacade.cs`)

Do NOT:
- Put substantial business logic in facades
- Bind directly to vendor SDKs
- Let facades grow into god-classes
- Let internal services depend on facades

### Service Rules

Services are internal implementation classes.

Rules:
- Each service expresses one clear capability
- Use standard .NET return types and exceptions (NOT `Res<T>`)
- Only consumed by Facades and other Services within the same module
- External modules depend on `Abstractions/` interfaces, not Services directly

Recommended naming:
- `KnowledgeBaseService`, `DocumentIndexingService`, `KnowledgeSearchService`

Avoid naming:
- `RAGManager`, `RAGHelper`, `RAGCoreService`

### Support Service Rules

Support services handle cross-cutting internal concerns.

Suitable types:
- `Registry` вЂ” registration and lookup
- `Resolver` вЂ” resolution and binding
- `Coordinator` вЂ” internal orchestration
- `Policy` вЂ” decision logic
- `Normalizer` вЂ” data normalization
- `Factory` вЂ” object creation

Examples: `ChunkerRegistry`, `RAGEmbeddingBindingResolver`, `RAGIndexStateCoordinator`

- A feature folder should NOT depend on another feature's internal `Services/` вЂ” use `Abstractions/` for cross-feature contracts

## UI Module Structure

### Core Rule

UI modules are pure presentation layers. They:

- Inject Facades from the infrastructure module directly
- Do NOT have their own service layer for data access
- Focus on components, page composition, state management, and localization

All UI modules use `UI{Name}/` feature directories вЂ” no flat root-level `Components/Services/Models`.

### UI Layer Responsibilities

| Folder | What belongs here | What does NOT belong here |
|--------|-------------------|--------------------------|
| `Pages/` | Route pages, page-level composition, lifecycle entry | Business orchestration, SDK calls |
| `UI{Name}/Components/` | Reusable Blazor components, partial UI composition | Page routing, business entry points |
| `UI{Name}/Dialogs/` | Dialog components | Non-dialog components |
| `UI{Name}/Models/` | ViewModels, DialogModels, display-only models | Infrastructure models (reuse from Facade Models) |
| `UI{Name}/State/` | Browser state, page state, session state, table state | Business orchestration |
| `UI{Name}/Support/` | Resolvers, formatters, coordinators, storage adapters | Data access (use Facade instead) |
| `Localization/` | Resource markers and localization JSON files | Business logic |

### UI Model Reuse Rule

UI modules should maximize reuse of Models from the infrastructure module's public `Models/` folder. Only create UI-specific models when:

- The view requires a shape that genuinely differs from any existing model
- The model is purely presentational (e.g., `DialogModel`, `ViewModel` with UI-only state)

Do NOT duplicate infrastructure models in the UI module.

### UI State / Support Rules

Not everything in the UI layer is a "service". Use precise folder placement:

| Type | Folder | Examples |
|------|--------|---------|
| Page/session state | `State/` | `ChatSessionStateManager`, `ChatSessionStorage` |
| Browser persistence | `State/` | `RAGBrowserState`, `TableStateStorage` |
| Orchestration helpers | `Support/` | `RAGBatchIndexCoordinator`, `RAGChunkViewCoordinator` |
| Resolution/lookup | `Support/` | `RAGMarkdownDocumentResolver` |
| Format conversion | `Support/` | `ChunkHighlightFormatter` |

## Mixed Module Structure

### When to Use

Mixed modules contain both infrastructure and UI code in a single project.

Suitable when:
- UI is lightweight
- Infrastructure and UI are tightly coupled in lifecycle
- No need for separate UI packaging

Not suitable when:
- UI pages keep growing
- Module evolves multiple independent UI sub-features
- Separate deployment is needed

When a mixed module outgrows its structure, upgrade to a separate `Monica.{Name}.UI` project.

## Naming Conventions

### Module Registration

| Component | Pattern | Example |
|-----------|---------|---------|
| Module class | `Module{Name}` | `ModuleRAG` |
| Options class | `Module{Name}Option` | `ModuleRAGOption` |
| Guide class | `Module{Name}Guide` | `ModuleRAGGuide` |
| Builder extension | `Mo.Add{Name}()` | `Mo.AddRAG()` |
| Module key | `EMoModuleKey.{Name}` | `EMoModuleKey.RAG` |

### Service Naming

| Type | Pattern | Examples |
|------|---------|---------|
| Facade | `{Feature}Facade` | `RAGFacade`, `AIChatFacade` |
| Business service | `{Feature}Service` | `DocumentIndexingService`, `KnowledgeSearchService` |
| Registry | `{Feature}Registry` | `ChunkerRegistry` |
| Resolver | `{Feature}Resolver` | `RAGEmbeddingBindingResolver` |
| Coordinator | `{Feature}Coordinator` | `RAGIndexStateCoordinator` |
| Provider | `{Vendor/Strategy}{Capability}Provider` | `FileDocumentIndexStateStore` |
| State manager | `{Feature}StateManager` | `ChatSessionStateManager` |

Avoid vague names: `Manager`, `Handler`, `Helper`, `Core`, `Base` (unless truly abstract base class).

### UI Naming

| Component | Pattern | Example |
|-----------|---------|---------|
| UI module | `Module{Name}UI` | `ModuleRAGUI` |
| UI folder | `UI{Name}/` | `UIChat/`, `UIRAG/` |
| Page | `UI{Name}Page` | `UIRAGManagePage` |
| Route URL | `/{name}-{action}` (kebab-case) | `/rag-manage`, `/ai-chat` |

## Prohibited Practices

- A single `XxxService` acting as facade + business logic + provider coordination + state recovery
- Internal services depending on Facades
- UI components depending on Providers directly
- Inventing custom folder names (`CoreCommunicationProvider/`, `BuildInMiddlewares/`, `ControlPlane/`, `WorkerPlane/`)
- Flat root-level `Components/Services/Models` in UI modules
- Duplicating infrastructure Models in UI modules when the same model can be reused
- Putting complex business logic directly in Minimal API handlers or Razor pages
- Using `Helpers/`, `Tools/` (except AI tools), `Common/`, `Misc/` as folder names вЂ” use `Utils/` instead

## Summary

Monica's unified module architecture follows these fixed patterns:

- **Infrastructure modules**: Feature-first organization, standard layers inside each feature, Facades as the public `Res<T>` entry point, `Internal/` sub-folders for private types
- **UI modules**: `UI{Name}/` feature directories, inject Facades directly, minimize UI-specific models, separate State and Support from Components
- **Mixed modules**: Infrastructure and UI coexist in one project but maintain strict layering boundaries
- **Bundled features**: `Features/` or root-level feature folders, each following the same layer convention

Three inviolable rules:

1. Facades are thin `Res<T>` orchestrators вЂ” not business logic dumping grounds
2. `Internal/` separates public contracts from implementation details
3. Providers are always a separate layer вЂ” never mixed with Services or UI
