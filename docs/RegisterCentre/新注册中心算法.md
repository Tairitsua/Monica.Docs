# 算法介绍

心跳周期：15s  （可配置）
reg:下线TTL：3倍心跳周期+1s。（可配置倍率）
leader:下线TTL：2倍心跳周期+1s。（可配置倍率）

挣扎周期：3s（可配置）
放弃挣扎阈值：5s（可配置，要高于挣扎周期）


1. 注册与心跳，其实初次注册也是心跳，因为inst_id肯定不会重复。

心跳周期执行：reg:{svc_name}:{inst_id}

心跳成功，记录下心跳成功时间，进入到检查/保持Leader逻辑。
心跳若失败，
检查上次心跳成功时间+reg下线ttl，如果已经小于当前时间，标记自己为孤立状态，根据设置，来决定快速下线或进入孤立模式

另外判断自己是否是Leader：
如果不是Leader，等待下次心跳。
如果自己是Leader，则进入/继续挣扎逻辑。

下次心跳时间增加±1秒内随机的抖动时间缓解redis压力

2. 检查/保持Leader

判断自己是否是Leader（内存）
a. 如果不是Leader，则进入监视当前Leader逻辑：
判断Key是否存在：leader:{svc_name}
若Key不存在，说明当前不存在Leader。进入到竞争Leader逻辑。
若Key存在，则什么都不做，继续心跳。

b. 如果自己是Leader，则进行续约Leader逻辑：
使用Etag确保保存成功：（初次Etag在竞争Leader逻辑成功后会获取并保存到内存）
Set leader With Etag:{svc_name} Value：{inst_id}  
成功后
Get New Etag 然后替换当前Etag并更新Leader成为时间。
若续约失败，
判断是由于网络原因还是由于当前redis中leader已不是自己而失败，如果是后者，则立即触发leader丢失事件（防御性编程）
如果是网络原因，则进入/继续挣扎逻辑。

3. 竞争Leader。
Set leader：{svc_name} Value：{inst_id}   
成功后
Get Value With Etag，确保Value和自己 inst_id 一致。
若一致：
竞争成功，标记自己为Leader，并记录下当前Leader成为时间。
若不一致：
说明竞争失败，结束当次心跳周期。

4. 挣扎逻辑：
该逻辑为减缓由于临时网络波动设计，并尽可能遏制存在的双Leader情况而设计。
每挣扎周期进行：
判断成为Leader时间+Leader TTL-放弃挣扎小于当前时间，则触发Leader丢失事件，终止相关Leader业务。
否则
进行心跳逻辑（心跳-> 成功后进行检查/保持Leader）如果成功，则退出挣扎。（内部如果又触发挣扎，则忽略）
继续新一轮循环。
