# 现状问题与重构目标

Monica 当前模块系统已经具备统一注册、依赖声明和分阶段执行能力，但它同时积累了几类明显的设计债。这些问题不是单个模块的局部实现问题，而是模块系统核心模型本身表达力不足。

## 当前最主要的问题

## 1. 依赖声明阶段与配置读取阶段混在一起

当前 `ClaimDependencies` 阶段会临时实例化模块，并允许模块在这个阶段做超过“声明依赖”范围的事情。结果是：

- 依赖声明不再是纯操作
- 模块作者很容易在错误阶段读取别的模块配置
- 某些需求不得不通过延迟解析或全局静态注册表绕过去

理想状态下，依赖声明阶段只做一件事：声明图结构。

## 2. `Guide` 同时承担太多职责

当前 `Guide` 既负责：

- 注册模块
- 写模块配置
- 声明依赖
- 发出生命周期请求
- 记录必须调用的方法

这导致 `Guide` 既像 DSL，又像注册器，又像配置写入器。职责过多以后，生命周期边界会变得模糊，模块作者也更难判断某个 API 在什么阶段是安全的。

## 3. `Option` 被当成了配置对象、运行时状态容器和跨模块写入目标

一个理想的 `Option` 应该只表达“用户可理解、可列举、可冻结的配置”。

但在现有系统里，`Option` 很容易被扩展成：

- 运行时状态存储
- 他模块的写入入口
- owner 模块的隐式聚合容器

这会让配置对象失去边界，最终变成“谁都能改一点”的共享结构。

## 4. 模块间配置读取缺少阶段约束

当前系统虽然提供了跨模块读取配置的能力，但“什么时候能读、哪里能读、哪些模块能读”没有被强约束表达。

结果是：

- 能不能读，依赖隐含的初始化顺序
- 读失败时通常是运行时异常，而不是 API 层面的约束
- 模块作者需要记住很多框架内部时序细节

## 5. 去重与冲突处理过于依赖字符串 key

当前不少注册动作通过字符串 key、`secondKey` 和去重规则来控制重复配置。这个模型能工作，但有明显问题：

- key 规则很难从 API 表面理解
- 冲突语义不稳定
- 去重发生在请求层，而不是 owner 的业务聚合层
- 很多本应由 owner 模块定义的“合并规则”被降级成了统一字符串去重

## 6. 全局静态注册表承担了过多协调职责

当前 `ModuleRegistry` 同时负责：

- 模块注册状态
- 生命周期推进
- 配置物化
- 执行请求
- 错误收集
- 部分跨模块查询

这会让系统过度依赖全局状态，也让很多能力只能以“静态查表”的方式暴露。

## 重构要达到的目标

## 1. 依赖声明必须纯化

依赖声明阶段只允许声明：

- 我依赖哪些模块
- 我需要哪些 capability
- 我暴露哪些 contribution / export 类型

这个阶段不允许读取他模块最终配置，不允许依赖运行时对象。

## 2. 配置必须先冻结，再共享

每个模块都应该先生成自己的 **Resolved Config**，然后再允许他模块读取。

换句话说：

- 写配置发生在 config build 阶段之前
- 跨模块读取发生在 config freeze 之后
- 被别人读取的一定是只读快照，而不是可变 option

## 3. Owner 模块拥有最终聚合权

像导航项、页面注册、根路由、Swagger UI 扩展、路由重定向这类能力，都应该由 owner 模块定义 contribution 合并规则。

外部模块不应直接改 owner 的 option。

## 4. 配置读取必须强类型、阶段安全、依赖可追踪

理想 API 需要同时回答三个问题：

- 读的是谁的配置
- 为什么你有权读它
- 现在这个阶段读它是否安全

因此新的读取方式应该：

- 强类型
- 依赖显式
- 只能读取直接依赖或显式声明可见的模块

## 5. 生命周期上下文必须比全局静态工具更重要

模块作者应主要通过强类型上下文完成工作，而不是通过全局静态入口查询系统状态。

上下文应显式区分：

- 自身配置
- 依赖配置
- 导出契约
- contribution 写入入口

## 6. 错误应该尽量前置

好的模块系统不是把错误“留到运行时自然爆炸”，而是尽量在下面这些阶段就给出明确错误：

- 依赖图构建阶段
- 配置冻结阶段
- contribution 聚合阶段
- 生命周期执行前的验证阶段

## 一句话目标

这轮重构的核心不是“让模块间更容易互相拿配置”，而是：

**让模块作者能在正确阶段，用正确方式，读取正确边界内的只读配置，并把跨模块协作尽量转成 owner-owned contribution。**
